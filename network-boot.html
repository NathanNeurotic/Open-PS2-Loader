Network Boot (SMB / UDP)

Stream your PS2 game library from a PC over the LAN. RiptOPL picks the network transport with a single selector — Device Settings → Network Protocol — offering Off / SMB / UDPFS / UDPFSBD / UDPBD. Because the PS2 has one Ethernet adapter, exactly one of these is ever active, so the selector is exclusive by construction. The three UDP transports each appear in RiptOPL as their own games list — with covers, Favourites, and per-game settings, just like a local drive — and always launch through Neutrino (there is no OPL built-in core fallback for them). SMB is the exception: it is a mounted file share served by OPL's own core, and needs no Neutrino.

ℹ One selector, Off by default
The old Network Boot ON/OFF toggle, the separate Net Boot Protocol picker, and the live SMB-vs-Network-Boot interlock are gone — a single Network Protocol selector replaces all three, and it defaults to Off (no network device). Each UDP transport needs a static PS2 IP and a matching PC-side server. Pick a protocol only when you have those pieces in place. Local devices (USB / internal HDD / MMCE) are independent and browse alongside whichever network protocol you choose.

The five Network Protocol choices

Set Settings → Device Settings → Network Protocol to one of five values. Because the PS2 has a single NIC, choosing one automatically excludes the others — no separate enable toggle, and no greying interlock to manage.

ChoiceWire protocolOn the PS2PC serverCore
Off — no network device (default) — —
SMB SMBv1 a mounted file share Samba / smbserver_opl.py OPL's own core (not Neutrino)
UDPFS UDPRDMA a filesystem (udpfs:) — loose ISOs in a folder udpfs_server.py -d <dir> / udpfsd Neutrino only
UDPFSBD UDPRDMA a block device (massN:) — a served disk image udpfs_server.py -b <image> / udpfsd Neutrino only
UDPBD SUDPBDv2 a block device (massN:) — a served disk image udpbd-server (external) Neutrino only

UDPBD is the original one: Rick Gaiser's SUDPBDv2 block device — to OPL it is just another block device (it mounts massN:), it has no <OPL> core backend, so its games always launch via Neutrino (-bsd=udpbd). UDPFS and UDPFSBD are two shapes of the newer UDPRDMA transport (see the comparison below). All three UDP transports have no DHCP client and need a static PS2 IP; if neutrino.elf is missing at launch, OPL warns and returns to the menu.

⚠ Emulator-validated only
The UDPRDMA path (both UDPFS and UDPFSBD shapes) is validated on emulator so far — real-PS2 confirmation is pending.

UDPFS vs UDPFSBD — two shapes of one UDPRDMA transport

UDPFS is Neutrino's newer network transport (Rick Gaiser's UDPRDMA), wire-incompatible with UDPBD's SUDPBDv2 (so it needs a different PC server). RiptOPL offers it in both shapes Neutrino supports, as two separate selector options:

UDPFS (filesystem)UDPFSBD (block)
IOP driverudpfs_ioman.irx → udpfs:udpfs_bd.irx → massN:
PC servesa folder of loose ISOsa FAT/exFAT disk image
Server commandudpfs_server.py -d <dir>udpfs_server.py -b <image>
Neutrino -bsd-bsd=udpfs (stock token)-bsd=udpfsbd (RiptOPL token)
Add a gamedrop a file in the foldermount the image, copy, unmount
Compression.zso/.cso/.chd via --enable-compression— (raw sectors only)

Both ride the same UDPRDMA transport and the same servers — pcm720/udpfsd (a prebuilt Go binary, serves both) or the bundled udpfs_server.py (Python 3); just pass -d for the filesystem or -b for the block image. A UDPBD-only udpbd-server (SUDPBDv2) will not talk to either.

  • UDPFS launches with the stock -bsd=udpfs, which loads Neutrino's shipped bsd-udpfs.toml (the FHI filesystem driver, udpfs_ioman / udpfs_fhi). Neutrino opens the game by name (-dvd=udpfs:<path>) — no massN: block device, no fragment list.
  • UDPFSBD launches with -bsd=udpfsbd, a RiptOPL-private token: Neutrino ships udpfs_bd.irx but no stock -bsd for it, so RiptOPL auto-places config/bsd-udpfsbd.toml into the bundled Neutrino folder. The private name is deliberate — it avoids colliding with stock's bsd-udpfs.toml (a different driver), so the block config and the stock filesystem config coexist on one install.
So why keep both?

They're the same protocol but a genuinely different workflow. UDPFS is the lower-friction, "modern" model (loose ISOs, drop-in, optional .zso/.cso/.chd compression) and matches how NHDDL and pcm720's server are designed. UDPFSBD reuses OPL's existing massN: block pipeline (a disk image the PS2 mounts with its own FAT/exFAT drivers) — handy if you already keep a block image or want the exact same shape as USB/HDD. Pick whichever fits how you store your games. Hardware note: the UDPRDMA path (both shapes) is validated on emulator only so far — real-PS2 confirmation is pending.

Where is udpfs_server.py?

The RiptOPL release ships Neutrino as a drag-and-drop neutrino/ folder in the release zip. Its udpfs_server/ directory contains udpfs_server.py — the PC daemon to run for either UDPFS shape (pass -d <dir> for the filesystem, -b <image> for the block image). UDPBD's udpbd-server is a separate project (israpps/udpbd-server, the canonical CI build) not bundled with RiptOPL — or use pcm720/udpfsd, a single Go binary that serves both UDPBD and UDPFS. RiptOPL's own PS2-Servers one-stop bundle collects the SMBv1, UDPBD, and UDPFS servers together.

Why does UDPFSBD need a private bsd-udpfsbd.toml when a stock bsd-udpfs.toml exists?

The -bsd token is just a config filename: at game boot Neutrino loads config/bsd-<token>.toml, and the toml's body picks the IOP driver. Stock Neutrino already ships a bsd-udpfs.toml whose body is the FHI filesystem driver (udpfs_ioman / udpfs_fhi, no i_bdm, registers no massN: block device). The UDPFS option uses that stock token as-is — -bsd=udpfs, opening the game -dvd=udpfs:<path> by name.

The UDPFSBD option needs the block-device body instead (udpfs_bd.irx, i_bdm → massN:) because that launch hands Neutrino -dvd=massN:. Neutrino ships udpfs_bd.irx but no built-in -bsd token for it, so RiptOPL auto-injects config/bsd-udpfsbd.toml into the bundled Neutrino folder at build time and emits -bsd=udpfsbd. The private name is deliberate — bsd-udpfsbd.toml will not collide with stock's bsd-udpfs.toml (a different driver), so the block config and the stock filesystem config coexist on one Neutrino install. If you assembled Neutrino yourself, copy neutrino/bsd-udpfsbd.toml from the RiptOPL source tree into your neutrino/config/ folder.

Requirements

  • Static PS2 IP address (UDP transports only). The UDPFS / UDPFSBD / UDPBD IOP modules have no DHCP client — they read the IP set in Settings → Network Config and use it directly. If DHCP is currently on there, RiptOPL warns you when you select a UDP protocol in Device Settings. (SMB can use DHCP or a static IP; it keeps its own address/port/share/credentials fields on the Network Config page.)
  • PC-side server running on the same LAN, speaking the protocol you selected — udpbd-server (israpps) for UDPBD, udpfs_server.py for UDPFS/UDPFSBD, or pcm720/udpfsd (a no-Python Go binary that serves both UDPBD and UDPFS). RiptOPL's own PS2-Servers bundles the SMBv1, UDPBD, and UDPFS servers.
  • Neutrino ELF for the UDP transports. UDPFS / UDPFSBD / UDPBD always launch via Neutrino; if neutrino.elf is not found at launch, RiptOPL warns and returns to the menu. The UDP feature ships its own bundled Neutrino (a ready-to-use neutrino/ folder in the release zip, pre-populated with the UDPFS config). See Neutrino Core. (SMB needs no Neutrino — it uses OPL's own core.)
  • Neutrino reads its own IP from a toml, not from OPL. When a UDP game boots, control hands to the external Neutrino, whose config/bsd-udpfsbd.toml / config/bsd-udpbd.toml hardcode ip=192.168.1.10. If your PS2's static IP differs, edit the ip= line in that toml (under the bundled neutrino/config/) to match — otherwise games list fine in OPL but fail to boot.
  • Game files laid out with standard OPL folders for the block shapes (UDPFSBD / UDPBD) and SMB: CD/, DVD/, ART/, CFG/, etc. — the same structure you would use on a USB drive. (UDPFS instead serves a flat folder of loose ISOs; drop the file in and the game appears.)

Selecting a network protocol

Set a static IP on the PS2 (UDP only)

For any of the three UDP transports, go to Settings → Network Config and set a fixed IP address, subnet mask, and gateway; turn DHCP off. The UDP IOP modules read this address at module-load time and have no way to acquire a lease later. (SMB can stay on DHCP if you prefer.)

Pick your Network Protocol

In Settings → Device Settings, set Network Protocol to one of SMB, UDPFS, UDPFSBD, or UDPBD (or Off to disable networking). This one selector replaces the old Network Boot toggle + Net Boot Protocol picker; because the PS2 has a single NIC, choosing one excludes the others automatically — there is nothing to grey out and no interlock to manage. Save settings. When a non-SMB protocol is selected, SMB's server / port / share / credentials fields on the Network Config page are hidden automatically.

Start the PC server

On the PC, run the server that matches your chosen protocol. Point it at the folder holding your games (a loose-ISO directory for UDPFS, a FAT/exFAT disk image for UDPFSBD/UDPBD, or a share for SMB).

# UDPBD (external server, SUDPBDv2)
udpbd-server /path/to/games

# UDPFS — filesystem, loose ISOs in a folder
python3 udpfs_server.py -d /path/to/isos
python3 udpfs_server.py -d /path/to/isos --enable-compression   # allow .zso/.cso/.chd

# UDPFSBD — block device, a served FAT/exFAT image
python3 udpfs_server.py -b /path/to/games.img

# UDPFS or UDPFSBD, no Python — pcm720/udpfsd (download a prebuilt binary)
udpfsd -d /path/to/isos            # filesystem (Linux/macOS)
udpfsd.exe -b C:\path\to\games.img # block image (Windows)

Boot the PS2 and find your games

RiptOPL loads the IOP module as soon as it starts. UDPFS mounts the udpfs: filesystem; UDPFSBD and UDPBD mount the served image as massN:. Your games appear in that protocol's own games list. Select a game and launch it — for the UDP transports, Neutrino is invoked automatically; SMB boots through OPL's own core.

Neutrino-only: no OPL core fallback (UDP transports)

UDPFS, UDPFSBD, and UDPBD have no embedded cdvdman backend, so they cannot use OPL's built-in EE core. Every UDP game launches via Neutrino, regardless of the per-game Loader Core setting. In the per-game Compatibility screen, the Loader Core selector is locked to Neutrino for these games — it cannot be changed to <OPL>. If neutrino.elf is missing when a game is launched, OPL shows a warning and returns to the menu; it does not attempt an OPL-core fallback.

SMB is different: it is served by OPL's own core, needs no Neutrino, and behaves like the classic OPL network share. See SMB (network share).

ℹ Per-game settings still work
UDP-transport games support the full per-game settings screen in its Neutrino-aware mode: compat modes 1/2/3/5 (mapped to -gc), mode 7 (fix IOP buffer overrun, -gc=7), the Neutrino Video picker (-gsm), VMC, and free-text Neutrino Launch Args. OPL-core-only panels (GSM, Cheats, PADEMU, OSD Language) show a "not used with the Neutrino core" notice. See Per-game Settings and Neutrino Core for details.

One NIC, one selector — no interlock to manage

The PS2 has a single Ethernet adapter (SMAP), and every network transport — SMB, UDPFS, UDPFSBD, UDPBD — loads a SMAP driver on it. Only one can own the adapter at a time, so RiptOPL exposes them as mutually exclusive values of one selector rather than independent toggles. This replaces the old model, where a separate Network Boot toggle and an Ethernet (SMB) mode had to be greyed against each other by a live interlock. Now you simply choose the one you want — switching between SMB and a UDP transport is just changing the Network Protocol value and saving.

Local devices — USB, MX4SIO, iLink, internal HDD (APA / exFAT), and MMCE — are entirely independent of this selector and browse alongside whichever network protocol you pick (or none).

Config file keys

These values are stored in settings_riptopl.cfg.

KeyTypeDefaultMeaning
network_protocolint (0–4)0Authoritative selector. 0=Off, 1=SMB, 2=UDPFS, 3=UDPFSBD, 4=UDPBD.
enable_udpbdbool (0/1)0Legacy back-compat shadow (superseded by network_protocol). Still dual-written: 1 whenever the protocol is UDPFSBD or UDPBD.
net_boot_protocolint (0/1)0Legacy back-compat shadow (superseded). 0 = UDPBD, 1 = UDPFS(BD). Only meaningful when the legacy enable_udpbd=1.
eth_modeint—Legacy back-compat shadow of the old ETH/SMB start mode; the unified selector supersedes it. Still written for older readers.

RiptOPL writes the legacy enable_udpbd / net_boot_protocol / eth_mode keys alongside the new one (a dual-write) purely so older builds can still read the config; network_protocol is the source of truth. The PS2 IP address used by the UDP IOP modules is read from the standard Network Config fields (the same ones SMB uses); there are no protocol-specific network keys.

Game folder layout on the server

For the block shapes (UDPFSBD / UDPBD) and SMB, the PC server exports a FAT/exFAT volume the PS2 reads with the same folder conventions as a USB drive:

mass0:/  (device root — folders sit here directly by default)
  CD/          ← PS2 CD-ROM images (.iso)
  DVD/         ← PS2 DVD images (.iso)
  ART/         ← cover art (same naming as USB)
  CFG/         ← per-game config files
  VMC/         ← virtual memory cards (optional)
  POPS/        ← PS1 POPS firmware (for VCD, if used)
  POPSTARTER/  ← POPSTARTER.ELF (for VCD, if used)
  APPS/        ← homebrew ELF files (optional)

The UDPFS filesystem shape is simpler: point udpfs_server.py -d (or udpfsd -d) at a plain folder of ISOs. Drop a file in and it appears in the UDPFS games list — no massN:, no fragment list, and optional .zso/.cso/.chd compression when the server runs with --enable-compression.

Can I use ZSO or UL-format images over the UDP transports?

It depends on the shape. UDPFS (filesystem) supports transparent .zso/.cso/.chd compression when the server is started with --enable-compression — Neutrino opens the file by name and the server decompresses on the fly. The block shapes (UDPFSBD / UDPBD) serve raw sectors only, and USB Extreme split (.ul) images require the OPL built-in core, which none of the UDP transports use — so on the block shapes, only standard ISO images work.

Troubleshooting

SymptomLikely causeFix
No network games list appears Network Protocol is Off, or the IOP module failed to load (no network link at boot). Check Device Settings → Network Protocol is set to your chosen transport (not Off). Verify the PS2 has a physical Ethernet link before power-on.
"needs a static PS2 IP" warning when selecting a UDP protocol DHCP is enabled in Network Config. Go to Settings → Network Config, disable DHCP, and enter a static IP.
List appears but game list is empty PC server not running, wrong IP, or the server speaks a different protocol than selected. Confirm the server is running. Confirm the PS2 IP in Network Config matches the server's expected client address. Confirm the Network Protocol value matches the server binary and mode (-d for UDPFS, -b for UDPFSBD, udpbd-server for UDPBD).
Game launches but immediately crashes or freezes neutrino.elf not found, wrong ip= in the Neutrino toml, or server disconnect mid-launch. Verify neutrino.elf exists (bundled neutrino/ folder or the custom Neutrino ELF Path). Check the ip= line in neutrino/config/bsd-udpfsbd.toml / bsd-udpbd.toml matches your PS2's static IP. Keep the PC server running throughout play.
UDPFSBD game fails but UDPBD works (or vice-versa) bsd-udpfsbd.toml missing from the Neutrino config folder, or the server is in the wrong mode for the shape you selected. Use the bundled Neutrino folder from the RiptOPL release (it contains config/bsd-udpfsbd.toml pre-injected). If you installed Neutrino manually, copy neutrino/bsd-udpfsbd.toml from the RiptOPL source into neutrino/config/. Make sure udpfs_server.py runs with -b for UDPFSBD (block) or -d for UDPFS (filesystem).
