Network Boot (SMB / UDP)

Stream your PS2 game library from a PC over the LAN. RiptOPL picks the network transport with a single selector — Device Settings → Network Protocol — offering four choices: Off / SMB / UDPFS / UDPBD. Because the PS2 has one Ethernet adapter, exactly one of these is ever active, so the selector is exclusive by construction. UDPFS is the modern network-boot protocol (Rick Gaiser's UDPRDMA transport); UDPBD is the older SUDPBDv2 protocol, kept for users still running the udpbd-server. Both appear in RiptOPL as their own games list — with covers, Favourites, and per-game settings, just like a local drive — and always launch through Neutrino (there is no OPL built-in core fallback for them). SMB is the exception: it is a mounted file share served by OPL's own core, and needs no Neutrino.

ℹ One selector, Off by default
The old Network Boot ON/OFF toggle, the separate Net Boot Protocol picker, and the live SMB-vs-Network-Boot interlock are gone — a single Network Protocol selector replaces all three, and it defaults to Off (no network device). UDPFS needs a static PS2 IP and a matching PC-side server. Pick a protocol only when you have those pieces in place. Local devices (USB / internal HDD / MMCE) are independent and browse alongside whichever network protocol you choose.

The four Network Protocol choices

Set Settings → Device Settings → Network Protocol to one of four values. Because the PS2 has a single NIC, choosing one automatically excludes the others — no separate enable toggle, and no greying interlock to manage.

ChoiceWire protocolOn the PS2PC serverCore
Off — no network device (default) — —
SMB SMBv1 a mounted file share Samba / smbserver_opl.py OPL's own core (not Neutrino)
UDPFS UDPRDMA a games source served over UDP — a filesystem (udpfs:, loose ISOs) or a block device (massN:, disk image), set by UDPFS Access below udpfsd (-fsroot Files or -bdpath Image) Neutrino only
UDPBD SUDPBDv2 a served disk image mounted as massN: udpbd-server Neutrino only

UDPFS is the one modern network-boot protocol (Rick Gaiser's UDPRDMA transport). It comes in two shapes, chosen with the UDPFS Access sub-setting (see the comparison below): Files (the udpfs_ioman filesystem, loose ISOs) or Image (the udpfs_bd block device, a served disk image). It has no DHCP client and needs a static PS2 IP; if neutrino.elf is missing at launch, OPL warns and returns to the menu.

ℹ UDPBD vs UDPFS — pick by your PC server
UDPBD (the older SUDPBDv2 udpbd.irx protocol) is a first-class fourth choice, not retired — it is kept because SUDPBDv2 is wire-incompatible with UDPRDMA (UDPBD speaks port 0xBDBD via udpbd-server; UDPFS speaks 0xF5F6 via udpfsd), so users still running the old udpbd-server must be able to keep selecting it. A config saved as UDPBD stays UDPBD — RiptOPL never silently migrates you off it. If you are setting up fresh, prefer UDPFS (it also serves loose ISOs and transparent .zso/.cso/.chd — see below).
⚠ Emulator-validated only
The UDPRDMA path (both Files and Image modes) is validated on emulator so far — real-PS2 confirmation is pending.

UDPFS Access — Files vs Image

When UDPFS is selected, a second sub-setting UDPFS Access appears with two values — Files (default) and Image. They are the same UDPFS protocol in two shapes: Files is the udpfs_ioman filesystem (udpfs:), Image is the udpfs_bd block device (massN:).

Files (default)Image
IOP driverudpfs_ioman.irx → udpfs: (filesystem)udpfs_bd.irx → massN: (block device)
PC servesa folder containing CD/ and DVD/ subfolders of ISOs (the standard OPL layout — OPL lists from <served>/CD + <served>/DVD, not the folder root)a FAT/exFAT disk image
Server commandudpfsd -fsroot <dir>udpfsd -bdpath <image>

Compressiontransparent .zso/.cso/.chd— (raw sectors only)

Both ride the same UDPRDMA transport and the same server — pcm720/udpfsd, a single prebuilt Go binary (no Python) that serves both; just pass -fsroot <dir> for Files (the filesystem) or -bdpath <image> for Image (the block image). A UDPBD-only udpbd-server (SUDPBDv2) will not talk to either. Because both backends bind the same UDPRDMA port and the IOP can load only one per boot, you can't run both at once — Files vs Image is a config-time choice, and switching between them needs an OPL restart to take effect (OPL shows the usual restart-to-apply notice).

  • Files launches with the stock -bsd=udpfs, which loads Neutrino's shipped bsd-udpfs.toml (the FHI filesystem driver, udpfs_ioman / udpfs_fhi). Neutrino opens the game by name (-dvd=udpfs:<name>) — no massN: block device, no fragment list. On the nightly Neutrino the -bsd is auto-detected.
  • Image launches with -bsd=udpfsbd, a RiptOPL-private token: Neutrino ships udpfs_bd.irx but no stock -bsd for it, so RiptOPL auto-places config/bsd-udpfsbd.toml into the bundled Neutrino folder. The private name is deliberate — it avoids colliding with stock's bsd-udpfs.toml (a different driver), so the block config and the stock filesystem config coexist on one install.
So why offer both Files and Image?

They're the same protocol but a genuinely different workflow. Files is the lower-friction, "modern" model (loose ISOs, drop-in, transparent .zso/.cso/.chd compression) and matches how NHDDL and pcm720's server are designed. Image reuses OPL's existing massN: block pipeline (a disk image the PS2 mounts with its own FAT/exFAT drivers) — handy if you already keep a block image or want the exact same shape as USB/HDD. Pick whichever fits how you store your games. (If you already run the old udpbd-server, that's a separate UDPBD choice — see above — not a UDPFS mode.) Hardware note: the UDPRDMA path (both modes) is validated on emulator only so far — real-PS2 confirmation is pending.

Which PC-side UDPFS server do I run?

The RiptOPL release ships Neutrino as a drag-and-drop neutrino/ folder in the release zip (neutrino.elf + config/ + modules/), but it does not bundle a UDPFS PC server. Get pcm720/udpfsd — a single prebuilt Go binary (no Python) that serves both modes: pass -fsroot <dir> for Files and -bdpath <image> for Image. RiptOPL's own PS2-Servers one-stop bundle collects the SMBv1 and UDPFS servers together. The old udpbd-server (SUDPBDv2) does not work with UDPFS — it is a different protocol served by the separate UDPBD Network Protocol choice.

Why does UDPFS Image need a private bsd-udpfsbd.toml when a stock bsd-udpfs.toml exists?

The -bsd token is just a config filename: at game boot Neutrino loads config/bsd-<token>.toml, and the toml's body picks the IOP driver. Stock Neutrino already ships a bsd-udpfs.toml whose body is the FHI filesystem driver (udpfs_ioman / udpfs_fhi, no i_bdm, registers no massN: block device). The Files mode uses that stock token as-is — -bsd=udpfs, opening the game -dvd=udpfs:<name> by name.

The Image mode needs the block-device body instead (udpfs_bd.irx, i_bdm → massN:) because that launch hands Neutrino -dvd=massN:. Neutrino ships udpfs_bd.irx but no built-in -bsd token for it, so RiptOPL auto-injects config/bsd-udpfsbd.toml into the bundled Neutrino folder at build time and emits -bsd=udpfsbd. The private name is deliberate — bsd-udpfsbd.toml will not collide with stock's bsd-udpfs.toml (a different driver), so the block config and the stock filesystem config coexist on one Neutrino install. If you assembled Neutrino yourself, copy neutrino/bsd-udpfsbd.toml from the RiptOPL source tree into your neutrino/config/ folder.

Requirements

  • Static PS2 IP address (UDPFS only). The UDPFS IOP modules (both Files and Image) have no DHCP client — they read the IP set in Settings → Network Config and use it directly. If DHCP is currently on there, RiptOPL warns you when you select UDPFS in Device Settings. (SMB can use DHCP or a static IP; it keeps its own address/port/share/credentials fields on the Network Config page, hidden automatically when UDPFS is selected.)
  • PC-side UDPFS server running on the same LAN, matching the UDPFS Access mode you chose. RiptOPL does not bundle one — get pcm720/udpfsd (a single prebuilt Go binary, no Python, that serves both -fsroot Files and -bdpath Image). RiptOPL's own PS2-Servers bundles the SMBv1 and UDPFS servers. The old udpbd-server (SUDPBDv2) does not work with UDPFS — it belongs to the separate UDPBD Network Protocol choice.
  • Neutrino ELF for UDPFS. UDPFS (Files and Image) always launches via Neutrino; if neutrino.elf is not found at launch, RiptOPL warns and returns to the menu. The UDPFS feature ships its own bundled Neutrino (a ready-to-use neutrino/ folder in the release zip, pre-populated with the UDPFS config). See Neutrino Core. (SMB needs no Neutrino — it uses OPL's own core.)
  • Neutrino reads its own IP from a toml, not from OPL. When a UDPFS game boots, control hands to the external Neutrino, which reads the toml for the active mode — Files uses the stock config/bsd-udpfs.toml; Image uses the bundled config/bsd-udpfsbd.toml. Both hardcode ip=192.168.1.10. If your PS2's static IP differs, edit the ip= line in whichever toml is active (under the bundled neutrino/config/) to match — otherwise games list fine in OPL but fail to boot.
  • Game files laid out with standard OPL folders for the Image mode and SMB: CD/, DVD/, ART/, CFG/, etc. — the same structure you would use on a USB drive. UDPFS Files also expects the standard layout: its served directory must contain CD/ and DVD/ subfolders (OPL lists from <served>/CD + <served>/DVD, never loose ISOs at the served root); drop an ISO into CD/ or DVD/ and the game appears.

Selecting a network protocol

Set a static IP on the PS2 (UDPFS only)

For UDPFS (either access mode), go to Settings → Network Config and set a fixed IP address, subnet mask, and gateway; turn DHCP off. The UDPFS IOP modules read this address at module-load time and have no way to acquire a lease later. (SMB can stay on DHCP if you prefer.)

Pick your Network Protocol (and UDPFS Access)

In Settings → Device Settings, set Network Protocol to SMB, UDPFS, or UDPBD (or Off to disable networking). When you pick UDPFS, a UDPFS Access sub-setting appears — choose Files (default) or Image (a served disk image). This one selector replaces the old Network Boot toggle + Net Boot Protocol picker; because the PS2 has a single NIC, choosing one excludes the others automatically — there is nothing to grey out and no interlock to manage. Save settings. When UDPFS is selected, SMB's server / port / share / credentials fields on the Network Config page are hidden automatically. Switching between Files and Image needs an OPL restart to take effect (only one UDPFS backend can bind the transport per boot).

Start the PC server

On the PC, run pcm720/udpfsd in the mode that matches your chosen access mode. Point it at the folder holding your games (a directory with CD/ and DVD/ subfolders for Files, a FAT/exFAT disk image for Image), or a share for SMB.

# UDPFS Files — a folder with CD/ and DVD/ subfolders of ISOs
udpfsd -fsroot /path/to/games            # Files   (Linux/macOS)
udpfsd.exe -fsroot C:\path\to\games      # Files   (Windows)

# UDPFS Image — a served FAT/exFAT disk image
udpfsd -bdpath /path/to/games.img        # Image   (Linux/macOS)
udpfsd.exe -bdpath C:\path\to\games.img  # Image   (Windows)

Boot the PS2 and find your games

RiptOPL loads the IOP module as soon as it starts. UDPFS Files mounts the udpfs: filesystem; UDPFS Image mounts the served image as massN:. Your games appear in the UDPFS games list. Select a game and launch it — Neutrino is invoked automatically; SMB boots through OPL's own core.

Neutrino-only: no OPL core fallback (UDPFS)

UDPFS (both Files and Image) has no embedded cdvdman backend, so it cannot use OPL's built-in EE core. Every UDPFS game launches via Neutrino, regardless of the per-game Loader Core setting. In the per-game Compatibility screen, the Loader Core selector is locked to Neutrino for these games — it cannot be changed to <OPL>. If neutrino.elf is missing when a game is launched, OPL shows a warning and returns to the menu; it does not attempt an OPL-core fallback.

SMB is different: it is served by OPL's own core, needs no Neutrino, and behaves like the classic OPL network share. See SMB (network share).

ℹ Per-game settings still work
UDPFS games support the full per-game settings screen in its Neutrino-aware mode: compat modes 1/2/3/5 (mapped to -gc), mode 7 (fix IOP buffer overrun, -gc=7), the Neutrino Video picker (-gsm), VMC, and free-text Neutrino Launch Args. OPL-core-only panels (GSM, Cheats, PADEMU, OSD Language) show a "not used with the Neutrino core" notice. See Per-game Settings and Neutrino Core for details.

One NIC, one selector — no interlock to manage

The PS2 has a single Ethernet adapter (SMAP), and every network transport — SMB and UDPFS — loads a SMAP driver on it. Only one can own the adapter at a time, so RiptOPL exposes them as mutually exclusive values of one selector rather than independent toggles. This replaces the old model, where a separate Network Boot toggle and an Ethernet (SMB) mode had to be greyed against each other by a live interlock. Now you simply choose the one you want — switching between SMB and UDPFS is just changing the Network Protocol value and saving. (The same one-transport-per-boot rule is why UDPFS Access = Files vs Image is a config-time choice, applied on the next restart.)

Local devices — USB, MX4SIO, iLink, internal HDD (APA / exFAT), and MMCE — are entirely independent of this selector and browse alongside whichever network protocol you pick (or none).

Config file keys

These values are stored in settings_riptopl.cfg.

KeyTypeDefaultMeaning
network_protocolint (0–4)0Authoritative selector. User-facing: 0=Off, 1=SMB, 2=UDPFS (Access = Files). 3=UDPFSBD is the internal value for UDPFS Access = Image — the picker maps both 2 and 3 onto the single "UDPFS" menu entry, and the Files/Image sub-setting picks between them. 4=UDPBD is a first-class value for the retained UDPBD (SUDPBDv2) choice: it is preserved as-is on load, never migrated, so users still on the old udpbd-server keep it.
enable_udpbdbool (0/1)0Legacy back-compat shadow (superseded by network_protocol). Still dual-written: 1 whenever the protocol resolves to UDPFSBD (Image).
net_boot_protocolint (0/1)0Legacy back-compat shadow (superseded). Derived from the selector; only meaningful when the legacy enable_udpbd=1. Selecting UDPBD sets enable_udpbd=1 with the classic UDPBD encoding, so a fresh UDPBD config still produces the legacy UDPBD shadow values.
eth_modeint—Legacy back-compat shadow of the old ETH/SMB start mode; the unified selector supersedes it. Still written for older readers.

There is no separate config key for the UDPFS Access (Files/Image) sub-setting — it is folded into network_protocol itself (2 = Files, 3 = Image). RiptOPL writes the legacy enable_udpbd / net_boot_protocol / eth_mode keys alongside the new one (a dual-write) purely so older builds can still read the config; network_protocol is the source of truth. The PS2 IP address used by the UDPFS IOP modules is read from the standard Network Config fields (the same ones SMB uses); there are no protocol-specific network keys.

Game folder layout on the server

For the Image mode (UDPFS Access = Image) and SMB, the PC server exports a FAT/exFAT volume the PS2 reads with the same folder conventions as a USB drive:

mass0:/  (device root — folders sit here directly by default)
  CD/          ← PS2 CD-ROM images (.iso)
  DVD/         ← PS2 DVD images (.iso)
  ART/         ← cover art (same naming as USB)
  CFG/         ← per-game config files
  VMC/         ← virtual memory cards (optional)
  POPS/        ← PS1 POPS firmware (for VCD, if used)
  POPSTARTER/  ← POPSTARTER.ELF (for VCD, if used)
  APPS/        ← homebrew ELF files (optional)

The UDPFS Files mode uses the same CD/ + DVD/ layout, just without a disk image: point udpfsd -fsroot at a folder that contains CD/ and DVD/ subfolders of ISOs (OPL lists from <served>/CD + <served>/DVD, not the served root). Drop an ISO into CD/ or DVD/ and it appears in the UDPFS games list — no massN:, no fragment list, and transparent .zso/.cso/.chd compression.

Can I use ZSO or UL-format images over UDPFS?

It depends on the access mode. Files (the filesystem) supports transparent .zso/.cso/.chd compression — Neutrino opens the file by name and the server decompresses on the fly. The Image mode serves raw sectors only, and USB Extreme split (.ul) images require the OPL built-in core, which UDPFS does not use — so in Image mode, only standard ISO images work.

Troubleshooting

SymptomLikely causeFix
No network games list appears Network Protocol is Off, or the IOP module failed to load (no network link at boot). Check Device Settings → Network Protocol is set to your chosen transport (not Off). Verify the PS2 has a physical Ethernet link before power-on.
"needs a static PS2 IP" warning when selecting UDPFS DHCP is enabled in Network Config. Go to Settings → Network Config, disable DHCP, and enter a static IP.
List appears but game list is empty PC server not running, wrong IP, or the server is in a different mode than the UDPFS Access you selected. Confirm the server is running. Confirm the PS2 IP in Network Config matches the server's expected client address. Confirm the UDPFS Access mode matches the server flag (-fsroot for Files, -bdpath for Image).
Game launches but immediately crashes or freezes neutrino.elf not found, wrong ip= in the Neutrino toml, or server disconnect mid-launch. Verify neutrino.elf exists (bundled neutrino/ folder or the custom Neutrino ELF Path). Check the ip= line in the active mode's toml (neutrino/config/bsd-udpfs.toml for Files, bsd-udpfsbd.toml for Image) matches your PS2's static IP. Keep the PC server running throughout play.
UDPFS Image game fails but Files works (or vice-versa) For Image: bsd-udpfsbd.toml missing from the Neutrino config folder, or the server is in the wrong mode for the UDPFS Access you selected. (Switching Files ↔ Image also needs an OPL restart.) Use the bundled Neutrino folder from the RiptOPL release (it contains config/bsd-udpfsbd.toml pre-injected). If you installed Neutrino manually, copy neutrino/bsd-udpfsbd.toml from the RiptOPL source into neutrino/config/. Make sure the server runs with -bdpath for Image (block) or -fsroot for Files (filesystem), and that you restarted OPL after switching UDPFS Access.
Existing UDPBD setup stopped working after updating The Network Protocol selector is set to something other than UDPBD (UDPBD is wire-incompatible with UDPFS, so a UDPFS pick won't talk to udpbd-server). Set Device Settings → Network Protocol back to UDPBD and keep running the same udpbd-server on the PC — a config saved as UDPBD is preserved as-is, never migrated. (Alternatively, switch to UDPFS and run udpfsd instead — but that is a different protocol and server.)
