## 1. Settings / cwd config (symptom 1)

### Root causes (all CONFIRMED by independent re-trace)

- **Read-before-drivers, on every non-MC boot.** `sysReset()`
  ([system.c:189](../src/system.c)) loads only
  iomanx/filexio/sio2man/mcman/mcserv/padman/usbd/â€¦â€” no `bdm.irx`, no
  `bdmfs_fatfs`, no block drivers, no `mmceman`. `init()` then does
  `configInit(gBootDir)` â†’ synchronous `_loadConfig()` ([opl.c:2316/2348](../src/opl.c))
  **before** any device support initializes. So `open("mass0:/APPS/settings_riptopl.cfg")`
  fails on a freshly reset IOP; defaults load; an immediate save fails
  ("Error saving settings"). A save after the user opens the device tab *succeeds* â€”
  but the next boot can't read it back, so settings never persist. PCSX2 masked this
  (`host:` is always mounted), which is how 8d6232f2 shipped "validated".
- **Dead fallback.** The legacy discovery / alternate-save gates
  ([opl.c:1217](../src/opl.c), [opl.c:1641](../src/opl.c)) require `gBootDir[0]=='\0'`,
  which essentially never happens (argv[0] is set by every real launcher, getcwd backup
  otherwise). `tryAlternateDevice` â€” the code that knows how to force-load drivers â€” is
  unreachable.
- **exFAT-HDD boots are doubly broken.** Form A: `argv[0]="ata0:/..."` is a
  launch-binding identity â€” never an openable filesystem â€” so read AND save fail
  forever. Form B: `argv[0]="mass0:/..."` needs `ata`-backed `massN:` mounted, which
  needs `hddLoadModules()` via `gEnableBdmHDD=1` â€” a flag stored in the unreachable
  config (circular; fork default is OFF, [opl.c:2288](../src/opl.c)). With a USB stick
  also attached, a save can silently land on the *stick*.
- **setBootDir edge cases** (medium): uLE APA-HDD paths (`hdd0:__common:pfs:/...`),
  backslash paths, and `host:C:\...` all produce unopenable or wrong boot dirs with no
  fallback.

### Fix design (verified against the code; branch `fix/bootdir-resolve-massn` @ `a245b607` is a start but INSUFFICIENT alone)

The existing unmerged commit `a245b607` resolves `ata0:`-style identities to `massN:`
at deferred-config-load time â€” correct direction, but it does **not** fix the
read-before-drivers hole (a `mass0:` boot dir is left untouched and still unreadable at
boot), and on `ps2dev:latest` the typed-devices delta (Â§5) would make
`bdmGetDeviceRootByType` return `ata0:` â€” a no-op rewrite. Full design:

1. **Classify** the boot prefix right after `setBootDir` (strip trailing digits from the
   `<device>:` token): `mc/host` â†’ ready; `mmce` â†’ needs mmceLoadModules;
   `mass` â†’ needs BDM (type unknown); `usb/ata/sdc/mx4sio/sd/ilink` â†’ needs BDM (typed);
   `hdd0:...pfs...` multi-colon and unknown â†’ UNRESOLVED. Normalize `\` â†’ `/`.
2. **Bring-up before the first read, after `ioInit`**: place it after `ioInit()`
   (opl.c:2322) and before `_loadConfig()` (opl.c:2348) â€” NOT before `configInit`,
   because `bdmLoadModules` queues block drivers via `ioPutRequest`, which silently
   drops requests before `ioInit` registers the handler (ioman.c:227). Load
   `bdmLoadModules()` + the classified block driver synchronously, **ignoring
   `gEnable*` defaults** (for ATA: `hddLoadModules()`; precedent: `miniInit`
   opl.c:2405-2413 and `checkLoadConfigBDMHDD` opl.c:1091).
3. **Resolve** with a bounded poll (~3 s USB / ~10 s ATA spin-up; factor the
   autoLaunchBDMGame loop at opl.c:2540 into a helper â€” its current bounds are much
   shorter): typed identity â†’ first `massN` whose driver matches
   (`USBMASS_IOCTL_GET_DRIVERNAME`/`GET_DEVICE_NUMBER`); untyped `massN` â†’ verify by
   probing `<candidate>:/<folder>/<elf-basename>` so launcher-vs-OPL slot renumbering
   can't land settings on the wrong stick. Rewrite only the device prefix
   (`ata0:/RIPTOPL` â†’ `mass1:/RIPTOPL`).
4. **UNRESOLVED after timeout** â†’ clear `gBootDir` (or add a `gBootDirResolved` flag
   consumed by the opl.c:1217/1641 gates) so legacy discovery re-arms â€” restores
   pre-8d6232f2 behavior for exotic launchers instead of failing forever.
5. **Reconcile after load**: if the loaded config disables the boot device's driver
   (`gEnableBdmHDD=0` while booted from ATA), force it on and `configSetInt` it
   (precedent opl.c:1097-1099).
6. **Save-side**: on write failure with a BDM boot dir, re-resolve once (hotplug may
   have renumbered slots) and retry. Never fall back to another device (PR #59 was
   rejected for that).
7. **Typed-root guard** (Â§5 dependency): the resolver must prefer legacy `massN:` roots
   even when latest-SDK typed devices make `ata0:` openable.

Bonus: the resolved `gBootDir` automatically fixes the POPSTARTER "Default (cwd)" tier
([vcdsupport.c:193](../src/vcdsupport.c)).

Also worth folding in (medium/low, same area): `configWriteMulti` counts unmodified
sets as successes (save can report success when the master write failed);
`guiHandleDeferedIO`'s 30 s bound counts FIFO **queue** time, so a save queued behind a
long list-refresh errors spuriously (start the clock when the request starts, or drain
list work like art is drained).

### Tests
HW matrix: boot from MC, MMCE, USB, MX4SIO, internal exFAT HDD (both argv[0] forms),

and `settings_riptopl.cfg` exists ONLY in the boot folder. PCSX2 regression: `host:`
boot still saves. MC/MMCE boots must not gain any BDM wait.

## 2. UDPFS (symptom 2)

All day-one design gaps (dd8aeddc-era); none SDK-sensitive; most PCSX2-testable.
In likely order of user impact:

1. **Connect-once, no reconnect â€” CRITICAL, CONFIRMED.** The Files variant (the
   default sub-mode) connects inside `AddDrv` at module load with a ~5 s discovery
   window racing PHY autonegotiation (smap needs a hard minimum ~3 s), and nothing ever
   retries: `udpfs_init_dev` is the only caller of `udpfs_core_init`
   ([udpfs_ioman.c:76-93](../modules/network/udpfs_ioman/src/udpfs_ioman.c)), and any
   later recv timeout latches `STATE_DISCONNECTED` forever. The EE side latches
   `udpfsIomanModLoaded` on *load*, not connect
   ([udpfssupport.c:53](../src/udpfssupport.c)). The reconnect watchdog 1ed7a176 added
   to `udpfs_bd` was never mirrored into `udpfs_ioman`. **Fix:** register the device
   unconditionally; lazy-connect/reconnect on open/dopen/getstat (the bound UDPRDMA
   socket is reusable â€” socket-once is preserved), or clone the udpfs_bd watchdog.
2. **CD/DVD layout vs "flat folder" docs â€” HIGH, CONFIRMED.** OPL scans only
   `udpfs:/CD` + `udpfs:/DVD` (`sbReadList`), while docs/NEUTRINO.md:137/165 tells
   Files users to serve a **flat folder of loose ISOs** â†’ guaranteed empty list.
   **Fix decision needed:** (a) fix the docs (CD/+DVD/ under the served root â€” verify
   udpfsd handles subdir opens first), or (b) true flat-share support, which requires
   udpfs-aware path construction end-to-end (`sbCreatePath` inserts `CD/`/`DVD/`
   unconditionally, so a naive root-scan would list games that then fail to launch).
3. **Launch-time IP mismatch â€” HIGH, CONFIRMED.** OPL's own chain uses `ps2_ip`
   (browse works), but Neutrino reads `ip=192.168.1.10` hardcoded in
   `bsd-udpfs.toml`/`bsd-udpfsbd.toml`. Boot fails whenever the configured PS2 IP â‰ 
   192.168.1.10 with an unedited toml â€” including the out-of-box default
   (192.168.0.10). **Fix:** rewrite/generate the active toml's `ip=` from `ps2_ip` at
   launch (the Neutrino-hosting device stays mounted via UNMOUNT_EXCEPTION); minimum: warn on mismatch.
4. **UDPBD retirement breaks working setups silently â€” HIGH, CONFIRMED.** Saved UDPBD
   configs fold to UDPFSBD (UDPRDMA, port 0xF5F6) but the user's udpbd-server
   (SUDPBDv2, 0xBDBD) never answers; the tab just never appears. **Fix:** one-time
   in-app notice when the fold fires ("switch PC server to `udpfsd -b`").
5. **Missing PC server â€” MEDIUM, CONFIRMED.** Docs + release notes reference a bundled
   `udpfs_server.py`; `pc/` has none and the zip packages none (and release notes still
   call UDPBD "the default"). **Fix:** vendor a server into `pc/udpfsserver/` (package
   like PC-SMB-Server) or strip every "bundled" claim; fix the release-notes heredoc.
6. **Asymmetric NIC guard â€” MEDIUM.** `ethLoadModules` doesn't check
   `udpfsGetModulesLoaded()` (the other two directions are symmetric) â†’ in-session
   UDPFSâ†’SMB switch can double-drive the SMAP. One-line guard.
7. **DHCP default + UDP transport â€” MEDIUM.** ministack has no DHCP; `ps2_ip` default
   is used silently when IP Type = DHCP (the default). Warn at tab-entry/boot, not only
   on the protocol-change dialog.
8. **Enum renumbering (2/4 swap between a5229759 and f4ce126b) â€” configs saved by that
   ~21-min window misread today** (saved UDPBD=2 now selects Files). Low exposure;
   at most a sanity-check against legacy shadow keys *before* first re-save.
9. `udpfs_bd` tears down the block device from inside its own read/write callback on IO
   failure (re-entrant teardown; possible slot renumbering). Defer to the watchdog thread.

Cleared: the bundled Neutrino (v1.8.0-47) **does** ship `bsd-udpfs.toml` +
`udpfs_fhi/udpfs_ioman/udpfs_bd` modules (verified against the actual release asset) â€”
still add a packaging assert since the bundle is re-fetched every publish.

## 3. Internal exFAT HDD â†’ VCD/POPSTARTER (symptom 3)

- **DEV9 power-off on the launch path â€” CRITICAL (PLAUSIBLE; final link is HW-only).**
  `bdmLaunchVcd` resolves the ELF, then `deinit(UNMOUNT_EXCEPTION, <bdm mode>)` â†’
  `moduleCleanup` calls `itemShutdown` for every non-selected support *without* the
  exception flag â†’ `hddShutdown` ([hddsupport.c:1025](../src/hddsupport.c)) decrements
  dev9 and powers the DEV9/ATA bus **off** â†’ `sysLaunchPopstarter` then `open()`s
  `mass<N>:POPS/POPSTARTER.ELF` on a dead bus â†’ elf-loader returns into deinit'd OPL =
  black screen. ISO launches are immune (embedded ee_core, no post-deinit read).
  PCSX2 masks it (emulated DEV9 power-off is inert). Refcount defect is sharper than
  one bad call: dev9InitCount only ever reaches 1 from the HDD side while hddShutdown
  decrements per call. **Fix options:** pair `sysShutdownDev9()` with
  `hddModulesLoadCount` reaching 0 (minimal), and/or never power DEV9 down on the
  launch path at all (POPSTARTER/ee_core reset the IOP anyway) â€” reserve power-off for
  real poweroff/exit.
- **Typed-devices SDK delta â€” CRITICAL on `ps2dev:latest` only (investigator-confirmed).**
  ps2sdk 2026-02-26 (`3061dbe2`, `5d64b0eb`) gives block devices a `path` and registers
  REAL `ata0:`/`usb0:`/`mx4sio0:`/`ilink0:` iomanX filesystems. RiptOPL's
  `bdmProbeTypedDeviceRoot` branch â€” previously self-defeating by design ("Dopen always
  fails") â€” now *succeeds* on latest builds, so `bdmGetDeviceRootByType(BDM_TYPE_ATA)`
  returns `ata0:/` and VCD/POPSTARTER/Neutrino paths get built on `ata0:`, violating
  the fork's own massN-throughout rule. **Fix:** prefer the legacy `mass<slot>:` root
  unconditionally in `bdmResolveDeviceRoot` (or gate typed roots behind a flag);
  keep launch-binding tokens as-is. NOTE: this invalidates the standing "typed roots
  never open" invariant â€” treat it as SDK-version-dependent from now on.
- **Picker "HDD (APA)" leg â€” HIGH, CONFIRMED.** `POPS_DEV_APA_HDD` resolves
  `pfs0:/POPS/POPSTARTER.ELF`; a launch from any non-HDD page then unmounts `pfs0:` in
  deinit before the ELF is read. **Fix:** allow deinit to except the ELF-holding
  device's mode too, or make hddShutdown honor the exception context, or skip the
  `pfs0:` leg for non-HDD launches (fall through to Default).
- **Silent BDMA equip fallback wipes working pairs â€” HIGH, CONFIRMED.** On ANY equip
  failure (missing `.ata` files, full MC, IO error), `vcdEnsureBdmaForLaunch` silently
  downgrades to FAT32, **deleting** `mc?:/POPSTARTER/usbd.irx`+`usbhdfsd.irx` (a
  working usbexfat pair from a previous session is collateral). No message on the
  launch path (settings-screen equip does surface diagnostics). **Fix:** surface the
  -4/-2/-3 diagnostic + abort the launch; never destroy an equipped pair as fallback.
- **gBDMPrefix desync â€” MEDIUM.** With a BDM prefix set, OPL scans
  `mass0:<prefix>/POPS` but POPSTARTER always reads `<root>/POPS` â†’ VCDs list, never
  boot. Scan device-root POPS regardless of prefix (or annotate).
- **Issue #56 open sub-report:** PS1-from-MMCE launch shows POPSTARTER debug
  `Cannot mount __common to pfs1` â€” POPSTARTER taking its APA/HDD path for an
  MMCE-hosted VCD. Needs one targeted trace of `mmceLaunchVcd` + which POPSTARTER
  variant the zip's `POPSTARTER/` folder actually contains (it's packaged from
  `popsmb/`, the SMB variant). Not yet root-caused.
